# 我的 PaiSmart 项目面试题（个人项目口吻，超详细版，可展开到 5 分钟）

说明：以下回答以“我自己做的项目”为视角，回答更细、可扩展成 5 分钟阐述。为了便于背诵，我保持结构清晰、内容分段。

## 后端面试题（重点）

### 项目总体

1. 请概述 PaiSmart 的整体架构与核心业务流程。
A（5分钟版）：这是我个人项目，我从一开始就按“前后端分离 + 多存储 + 异步任务”的思路设计。前端用 Vue3 负责交互，后端用 Spring Boot 提供 API 和 WebSocket。数据存储分层：MySQL 管元数据，Redis 做缓存和进度，MinIO 存文件本体，Elasticsearch 做文本与向量检索，Kafka 用于异步任务。这样分层的好处是职责清晰、性能稳定、扩展简单。

我重点实现的是文档从上传到可检索的全链路：用户分片上传→服务端合并→投递 Kafka→消费者解析→文本分块→向量化→写 ES。检索阶段我做了混合检索，先语义召回再关键词精排，同时做权限过滤，确保用户只能检索到允许访问的数据。对话模块走 WebSocket，实现流式输出；非实时操作全部用 HTTP。

在设计上，我关注的是“高并发可用”和“权限安全”。上传用 Redis bitmap 记录分片状态来保证断点续传；权限层面用 JWT + 组织标签做隔离；性能上用 Kafka 让重任务异步，避免阻塞用户请求。这些都是我在项目中亲自设计并落地的。

2. 文档上传到可检索的完整链路是什么？关键组件各自负责什么？
A（5分钟版）：上传的第一步是前端按固定大小分片并计算 fileMd5。后端 UploadController 接收分片请求，UploadService 将分片写入 MinIO，并在 Redis bitmap 里标记分片状态。我这样设计是为了断点续传和幂等。

分片全部到齐后调用 `/merge` 合并。后端用 MinIO 的 compose 接口合并成 `merged/{fileName}`，然后发送 Kafka 任务。FileProcessingConsumer 消费任务后下载合并文件。ParseService 用 Tika 解析并按“父块+子块”策略切分，写入 document_vectors。VectorizationService 调 Embedding API 生成向量，再通过 ElasticsearchService 批量写入 ES 索引。最后 HybridSearchService 能检索到这些文本块，并通过 ChatHandler 拼上下文喂给模型。

每个组件职责非常明确：上传相关负责存储和进度；消费者负责解析和向量化；检索服务负责召回和权限过滤；聊天服务负责上下文拼接和流式输出。我可以清楚讲出每一步的输入输出和异常处理方式。

3. 系统为何采用 Kafka 做异步处理？不用 Kafka 会有什么问题？
A（5分钟版）：解析和向量化是 CPU 与 IO 密集型任务，如果放在上传接口同步做，会导致接口超时，用户等待时间非常长。我在设计时就决定把这部分做成异步任务。Kafka 的优势是解耦、削峰、可重试。

异步带来的价值有三点：第一，上传接口快速返回，提高用户体验；第二，系统吞吐更高，可以同时处理很多上传请求；第三，失败任务可以重试或进入死信队列，可靠性更高。如果不用 Kafka，我就要自己实现任务队列和重试逻辑，成本高且不稳定。

我认为 Kafka 是这个项目能稳定运行的关键组件之一。它让系统具备工程化能力，而不仅仅是功能可用。

4. RAG 流程的关键步骤有哪些？上下文是如何构造的？
A（5分钟版）：RAG 的核心是“先检索，再生成”。我实现的流程是：先对用户问题生成向量，然后在 ES 做 KNN 召回，再用关键词 match 过滤，最后用 rescore 做精排。检索结果还要做权限过滤，确保用户只能看到自己或公开或组织内的内容。

上下文构造是很关键的步骤。我会对检索到的文本块进行截断，防止 prompt 过长；同时在每段前标记来源文件名和编号，便于模型理解来源。我还在 prompt 里设置规则，比如要求模型只用检索内容回答，减少幻觉。最终上下文和规则一起作为系统提示输入模型。

这样做的好处是回答有依据、内容可追溯，并且能控制模型输出质量。这个部分是我自己重点调试过的。

5. WebSocket 聊天与普通 HTTP 接口在这里的职责边界是什么？
A（5分钟版）：我把 WebSocket 只用于实时对话，因为模型输出是流式的，需要边生成边推送。HTTP 用于用户管理、文件上传、下载、检索、权限控制等请求，这些都适合请求-响应模式。

WebSocket 的好处是低延迟和实时性，尤其在对话场景能提升体验；HTTP 更可靠、易做幂等和错误处理。我在设计时就把这两类接口分开，避免混用导致复杂度上升。这样系统更清晰，也更容易排查问题。

### Spring Boot / Spring Security

1. JWT 的生成与校验流程是什么？Access Token 与 Refresh Token 各自作用是什么？
A（5分钟版）：我实现的是双 Token 体系。用户登录成功后，服务端生成 Access Token 和 Refresh Token。Access Token 有效期较短，用于业务请求鉴权；Refresh Token 有效期更长，用于 Access Token 过期时刷新。

请求进入后，JWT 过滤器先验证 Access Token 的签名与过期时间，合法则放行，并把 userId、角色等信息放入请求上下文。如果 Access Token 过期，前端调用刷新接口，服务端验证 Refresh Token 是否有效，然后签发新的 Access Token。这样既安全又保证体验。

我之所以采用双 Token，是为了避免长期有效的 Access Token 带来的风险，同时减少用户频繁登录。整体安全性与体验之间是一个平衡，我认为这个方案比较合理。

2. 如何实现 token 失效与黑名单？为什么需要 Redis 参与？
A（5分钟版）：JWT 是无状态的，签发之后无法主动失效，所以我引入 Redis 作为 token 状态存储。登录时把 tokenId 写入 Redis，带上用户信息与过期时间。登出时把 tokenId 加入黑名单或直接删除有效记录。校验时先查 Redis 是否有效，再判断黑名单。

Redis 的优势是速度快、跨实例共享、支持 TTL。这样即使后端多实例部署，也能统一 token 状态。如果不使用 Redis，就只能靠过期时间，无法主动注销或批量失效。我觉得 Redis 在这里是必要的。

3. SecurityFilterChain 中的权限配置如何保证资源安全？
A（5分钟版）：我在 SecurityConfig 里把接口分为匿名、普通用户、管理员。登录/注册和静态资源放行，上传、文档、对话等接口需要 USER 或 ADMIN，管理接口只允许 ADMIN。JWT 过滤器负责鉴权，OrgTag 过滤器负责组织权限。

这样设计保证了三层安全：身份验证、角色权限、组织权限。我还特别注意了接口路径一致性，避免因为路径配置错误导致权限漏洞。整体上是一套比较完整的安全策略。

4. `OrgTagAuthorizationFilter` 的职责是什么？如何避免权限绕过？
A（5分钟版）：这个过滤器用来做组织标签权限控制。它从 token 中解析 orgTags，把它们作为访问权限的依据。所有涉及文档、检索、下载的请求都必须经过这层校验。

为了避免绕过，我不会相信前端传来的 orgTag，而是用服务端解析的有效标签。对公开资源放行，对私有资源强制校验。如果路径漏配，就会有权限风险，所以我会保持配置与控制器路径一致。

5. 如果要对某些接口做细粒度权限（基于组织标签），有哪些实现方式？
A（5分钟版）：我会优先在数据层过滤，例如 MySQL 查询加 orgTag 条件，ES 查询加 filter。这样无论 controller 做不做校验，结果都安全。其次可以用 Spring Security 的注解或自定义权限表达式做更细控制。过滤器也可以把有效标签注入请求供业务层使用。我认为核心原则是“数据层强校验”，避免仅在 controller 层校验。

### MySQL

1. `file_upload`、`document_vectors`、`chunk_info` 三张表的数据关系是什么？
A（5分钟版）：`file_upload` 是文件元数据表，记录 file_md5、file_name、user_id、org_tag 等。`document_vectors` 存储解析后的文本块，每个文件对应多个 chunk，所以是 file_upload 对 document_vectors 的一对多。`chunk_info` 存分片记录，同样是 file_upload 对 chunk_info 一对多。三表通过 file_md5 关联。

这种设计的优势是查询简单，缺点是外键约束弱，因为用的是 file_md5 字符串。我在文档里也写了升级方案：改成 file_upload_id 作为外键，以加强一致性。

2. 文档删除时如何确保数据库与外部存储一致性？
A（5分钟版）：我采用最终一致性策略。删除时顺序是：先删除 ES 索引，防止检索到；再删除 MinIO 文件，防止下载；最后删除 MySQL 元数据和向量记录。每一步都会记录日志，失败也不会阻断后续步骤。之后可以通过补偿任务或定期扫描清理残留。

由于 ES 和 MinIO 不支持数据库事务，所以只能做补偿式一致性。我在设计时已经考虑了这一点。

3. 哪些查询是高频的？如何设计索引？
A（5分钟版）：高频查询包括：按 fileMd5 查文件、按 fileName + isPublic 下载、按 orgTag 或 userId 列表展示文件。索引设计包括：`file_upload(file_md5,user_id)`、`file_upload(file_name,is_public)`、`file_upload(org_tag,is_public)`。向量表用 `document_vectors(file_md5,chunk_id)`。对 conversations 用 `(user_id,timestamp)`。这样可以显著降低查询时间。

4. `users.org_tags` 用逗号存储有哪些隐患？如何规范化建模？
A（5分钟版）：逗号存储的问题是无法外键约束、查询复杂、索引无效。规范化方案是建一张 `user_org_tags` 表，存 user_id 和 tag_id，形成多对多关系。这样查询与权限校验更准确，也能保证标签一致性。缺点是多一次 join，但在可维护性上更强。我把它作为后续优化方向。

5. 在高并发上传场景下，如何避免重复插入或脏数据？
A（5分钟版）：我通过 fileMd5 + userId 的幂等检查避免重复插入，同时可以加数据库唯一约束。Redis bitmap 记录分片状态，避免重复上传分片。合并时核对分片数量，防止不完整合并。MinIO 也会检查分片是否存在。多层保障下能避免脏数据。

### Redis

1. Redis 在该系统中承担哪些职责？分别对应什么 Key 设计？
A（5分钟版）：我主要用 Redis 做四件事：上传进度、对话历史、token 状态、组织标签缓存。上传进度 key 形如 `upload:{userId}:{fileMd5}`，对话历史 `conversation:{id}`，当前会话 `user:{id}:current_conversation`。token 相关是 `jwt:valid:{tokenId}` 与 `jwt:blacklist:{tokenId}`。组织标签缓存是 `user:org_tags:{username}` 和 `user:effective_org_tags:{username}`。这样的 key 设计可以清晰分区并易于管理 TTL。

2. 上传进度为什么使用 bitmap？相比 set 或 list 的优劣？
A（5分钟版）：bitmap 占用极小，适合记录海量分片上传状态。判断某分片是否上传只需位运算，性能高。set/list 会存储大量分片索引，内存成本高。bitmap 的缺点是只能记录 0/1，但对进度场景足够。我认为这是比较合理的选择。

3. 对话历史存 Redis 的优势与风险？如何控制内存？
A（5分钟版）：优势是读写快，适合短期会话。风险是内存占用与数据不持久。我的做法是限制条数，比如最多 20 条，设置 TTL 比如 7 天。若需要持久化，可以把重要对话异步写 MySQL。这样既能保证性能又能控制内存。

4. Token 缓存的 TTL 如何设置？为什么要比 JWT 过期时间更长？
A（5分钟版）：TTL 设置为 JWT 过期时间 + 5 分钟缓冲，避免时钟漂移导致误判。这样用户体验更稳定，避免 token 刚好过期时系统判定错误。

5. 如何避免 Redis 缓存与数据库数据不一致？
A（5分钟版）：我采用“数据库为准”的原则，更新数据库后立即清理相关缓存。读取时缓存未命中再查库并回填，设置合理 TTL。对于组织标签变更，我会清理所有用户有效标签缓存，避免权限错误。

### MinIO / 文件存储

1. 分片上传与合并的存储路径设计是什么？为什么这样设计？
A（5分钟版）：我设计分片路径为 `chunks/{fileMd5}/{chunkIndex}`，合并文件路径为 `merged/{fileName}`。fileMd5 能唯一标识文件，chunkIndex 能按顺序拼接。合并文件单独存放，便于下载、预览和权限控制。这样的路径设计清晰、易维护，方便排错和清理。

2. 预签名 URL 的安全风险是什么？如何降低风险？
A（5分钟版）：风险是 URL 泄露导致越权下载。降低风险的方法包括：短 TTL、服务端权限校验后再生成、敏感文件用服务端代理下载。还可以配合日志审计，检测异常访问。这样能把风险控制在可接受范围。

3. 合并成功后清理分片的策略是否可靠？如何保证幂等？
A（5分钟版）：合并成功后清理分片是必须的，否则存储膨胀。删除操作如果失败会记录日志，不影响主流程。重复删除不会出错，所以是幂等的。分片清理失败可用定期任务清理。整体可靠性较高。

4. 如何处理大文件上传失败重试与断点续传？
A（5分钟版）：前端计算 fileMd5，分片上传时服务端用 Redis bitmap 记录已上传分片。失败后重新上传时查询 bitmap，只补传缺失分片。合并时核对分片数量，保证完整。这样大文件也能断点续传，降低失败成本。

5. 如果存储从 MinIO 换成 OSS/S3，需要改哪些模块？
A（5分钟版）：主要改 UploadService、DocumentService、FileProcessingConsumer 中的存储 SDK 调用。为了减少改动，我会抽象存储接口，提供 MinIO 与 OSS 两套实现。这样业务逻辑不变，只替换存储实现即可。

### Kafka / 异步任务

1. 文件处理任务消息体包含哪些信息？为什么需要这些字段？
A（5分钟版）：我定义任务包含 fileMd5、filePath、fileName、userId、orgTag、isPublic。filePath 用于下载文件，fileMd5 做幂等标识，fileName 便于日志和预览。userId、orgTag、isPublic 用于权限和索引字段写入。这样消费者能独立完成任务，不依赖额外查询。

2. 消费端失败如何处理？重试与死信队列策略是什么？
A（5分钟版）：消费者抛异常后 Kafka 会重试，超过次数进入死信队列。死信队列可以人工排查或自动补偿。这样不会丢失任务，也便于追踪问题。对可恢复错误（网络）会重试，对不可恢复错误需人工处理。

3. 为什么向量化和解析不放在上传接口同步完成？
A（5分钟版）：同步会导致上传超时和用户等待。我用 Kafka 异步处理，上传接口只负责合并和投递任务。这样接口更快，系统吞吐更高。

4. 如何保证任务不会丢失？如何保证消息处理幂等？
A（5分钟版）：Kafka 持久化+ACK 保证消息不丢失。消费者幂等通过 fileMd5 判断是否已处理。如果已处理则跳过或覆盖，避免重复写入。

5. Kafka 消费端性能瓶颈可能出现在什么地方？如何优化？
A（5分钟版）：瓶颈在文件下载、解析、向量化、ES 写入。优化方式包括扩容消费者、批量处理、提高解析效率、ES bulk 写入。还可拆分任务，进一步并行。

### Elasticsearch / 检索

1. 混合检索中 KNN 与 BM25 各自作用是什么？
A（5分钟版）：KNN 用语义向量召回，适合理解意图；BM25 用关键词匹配，保证准确性。组合后能兼顾语义和精度。

2. 为什么要做 rescore？它解决了什么问题？
A（5分钟版）：rescore 在召回结果中做精排，把更相关的内容排前面，减少噪声，提高回答质量。

3. 索引中 `isPublic`、`orgTag`、`userId` 的过滤逻辑是什么？
A（5分钟版）：我实现的规则是用户自己的 + 公开 + 组织标签匹配，确保权限隔离。

4. 结果召回数量如何选择？召回过大或过小的影响？
A（5分钟版）：召回过小会漏掉答案，过大性能差。我通常用 topK*30 再精排。

5. 如何处理 ES 中向量维度变化或模型升级？
A（5分钟版）：需要新建索引或新字段，并逐步迁移，保证兼容性。

### 文档解析与向量化

1. 使用 Apache Tika 解析文档的优缺点是什么？
A（5分钟版）：优点是支持多格式，缺点是大文件解析耗资源，部分格式不稳定。

2. 为什么采用父块 + 子块切分策略？
A（5分钟版）：父块保证流式解析不 OOM，子块保证语义检索效果。

3. 文本切分时如何兼顾语义与性能？
A（5分钟版）：按段落、按句子、按词分层切分，保证语义完整并控制长度。

4. HanLP 的作用是什么？如果分词失败如何降级？
A（5分钟版）：HanLP 提升中文分词质量，失败就降级为字符切分。

5. 向量化批量请求的大小如何权衡？
A（5分钟版）：批量越大吞吐越高但超时风险也高，需要结合 API 限制调优。

### WebSocket / 实时通信

1. WebSocket 鉴权是如何实现的？是否安全？
A（5分钟版）：当前用 URL path 传 token，后端解析鉴权，安全性一般但可用。后续可以改用 Header 增强安全。

2. 为什么需要内部 stop token？它解决了什么问题？
A（5分钟版）：防止恶意用户伪造 stop 指令中断生成，是我加的安全保护。

3. 流式响应如何与对话历史一致性同步？
A（5分钟版）：回答完成后再写入历史，保证记录完整。

4. WebSocket 连接断开时如何保证资源释放？
A（5分钟版）：关闭时清理 session 和缓存，避免内存泄露。

5. 如何限制并发连接，避免资源耗尽？

## 场景题（新增，详细）

1. 场景：用户上传成功但检索不到文档，你如何排查？
A（详细）：我会按链路逐步排查。先看 file_upload 是否有记录，确认合并是否成功；再看 Kafka 任务是否发送成功，检查消费者日志是否解析完成；然后查 document_vectors 是否有文本块；最后查 ES 索引是否写入并确认权限过滤是否过严。如果 ES 有数据但检索不到，重点检查 query 参数和 orgTag/userId 的过滤条件。我会用日志和数据库查询配合定位问题。

2. 场景：Kafka 消费堆积严重，系统延迟变大，你怎么处理？
A（详细）：我先看消费者是否存活、是否报错；然后看消费速度瓶颈是下载、解析还是向量化。短期可增加消费者实例提高吞吐，中长期可以优化解析和向量化批处理。必要时对任务做优先级划分，确保关键任务先处理。

3. 场景：用户频繁反映下载链接失效。
A（详细）：我先确认预签名 URL 的 TTL 设置是否过短，或者生成时间是否过早。其次排查是否权限校验失败导致生成失败。也可能是客户端缓存了过期 URL，我会建议前端在下载前重新请求链接。最后检查 MinIO 时钟同步，避免签名时间异常。

4. 场景：某用户能检索到不属于自己的文档。
A（详细）：这是权限漏洞，我会首先检查 ES 查询过滤条件，确认是否包含 userId/orgTag/isPublic 的过滤。其次检查 orgTagCache 是否正确更新，是否有缓存污染。还要确认 SecurityConfig 中路径是否漏配。修复方式是在数据层强制过滤，避免依赖前端。

5. 场景：分片上传到 80% 失败，用户重新上传后进度异常。
A（详细）：我会检查 Redis bitmap 状态是否正确，可能某些分片标记不一致。我会通过 status 接口返回已上传分片列表，前端只补传缺失分片。如果 Redis 数据损坏，我会通过 MinIO 实际分片列表修复进度，必要时重置 bitmap。

6. 场景：ES 搜索结果相关性差，用户反馈回答不准。
A（详细）：我会检查分块策略是否过长或过短，导致语义不完整。其次检查向量模型版本是否一致，维度是否匹配。还会检查 rescore 是否生效，是否需要调大召回数量或调整 BM25 权重。必要时重建索引并重新向量化。

7. 场景：Redis 内存占用过高，系统开始抖动。
A（详细）：先检查是否有 key 没有 TTL 或历史对话过多。清理无用缓存，缩短 TTL，限制对话条数。对大 key 进行拆分或压缩。必要时扩容 Redis 或使用 Redis 集群。

8. 场景：大模型响应慢，用户体验差。
A（详细）：我会先确认上下文是否过长，减少召回数量、截断内容。必要时做缓存，对重复问题直接返回。还可以更换更快的模型或增加模型服务实例。前端可先显示“正在生成”并支持停止，减少等待焦虑。

9. 场景：文件解析失败率升高。
A（详细）：先查看失败文件类型是否集中，可能是 Tika 对某些格式支持不稳定。可以对特定格式用专用解析器或限制上传类型。检查解析过程中内存占用是否过高，必要时调小 parentChunkSize 或增加机器内存。

10. 场景：用户反馈登录后经常被踢下线。
A（详细）：我会检查 JWT 过期时间与 Redis TTL 是否匹配，是否存在时钟偏差。检查 refresh token 流程是否正确，是否被误加入黑名单。还要排查是否有并发刷新导致 token 覆盖。必要时延长 Access Token 或优化刷新逻辑。

这样做的好处是避免非授权用户看到不该看到的页面，同时也提高了系统的灵活性。如果以后增加新角色或新页面，只需要更新配置即可。

A（5分钟版）：我在前端维护一份“路由-角色”映射表，根据后端返回的角色进行过滤，生成最终菜单和路由。

A（5分钟版）：我在请求拦截器中统一处理 401。流程是：先尝试刷新 token，如果刷新成功就重发原请求；如果刷新失败，就清空本地状态并跳转登录页。

为了避免多请求并发刷新，我使用“单例 Promise”机制：一旦正在刷新，其他请求等待结果，刷新成功后再继续。

这样可以保证用户体验连续性，不会因为 token 过期导致频繁跳转登录，同时也避免后端被大量刷新请求打爆。

为了兼容刷新页面的场景，我会在路由守卫中检测“是否已加载动态路由”。如果没有加载，就重新注入。这样刷新时也不会丢失路由。

同时我会维护一个“权限初始化完成”的状态，避免在路由尚未准备好时渲染页面，防止白屏或 404。

5. 路由守卫的关键逻辑是什么？

此外我会处理“刷新时路由丢失”的问题：如果当前路由不在已注册列表，会重新注入动态路由，然后再 `next`。

整体逻辑的目标是保证权限准确、体验顺畅、并且不让用户看到不该看到的页面。

### 请求与状态管理

1. 请求封装的核心逻辑是什么？
A（5分钟版）：我封装了统一的 Axios 实例，核心逻辑包括：统一 baseURL、超时设置、自动携带 token、统一解析响应结构和错误码。

请求拦截器负责注入 token，响应拦截器负责处理 `code/message/data` 结构。如果业务失败，我会统一弹出提示，并在必要时触发登出或刷新逻辑。

这样封装后，业务代码只需要关注成功的数据，不用每次都写错误处理和 token 逻辑，提高了整体可维护性。

2. Token 刷新机制如何避免并发刷新问题？
A（5分钟版）：我用一个全局的 `refreshPromise` 来控制刷新。只要刷新流程开始，后续请求不会再触发新的刷新，而是等待已有的 Promise 结果。

刷新成功后，所有等待的请求会自动继续执行；刷新失败则统一清空状态并跳转登录。这样可以避免“多请求同时刷新”的风暴问题。

此外我还会设置“最大刷新重试次数”，防止陷入死循环，保证用户体验和系统稳定性。

3. 后端返回非 200 时，前端如何统一处理？
A（5分钟版）：我在响应拦截器中统一处理错误码，按照错误类型做分级：登录失效跳转登录，业务错误提示 message，系统错误提示“稍后重试”。

同时我会把错误码映射成用户可理解的文案，避免出现“未知错误”这样的体验问题。必要时我会记录日志，方便排查。

这样可以保证前端所有接口的错误处理逻辑一致，也减少重复代码。

4. 前端如何组织 API 模块与类型定义？
A（5分钟版）：我按业务模块拆分 API 文件，比如 `authApi`、`uploadApi`、`chatApi`。每个模块中只包含与该业务相关的接口。

类型定义则放在统一的 `types/` 或 `models/` 目录，DTO 与 VO 分离，方便对接后端。接口方法返回统一的泛型响应类型，保证调用方拿到的数据有完整类型。

这样做能提高可读性和可维护性，也能在接口变更时快速定位修改点。

5. 如何保证状态与 UI 一致性？
A（5分钟版）：我用 Pinia 作为单一状态源，所有关键状态都通过 store 管理，比如登录态、当前会话、文件列表等。

UI 组件只订阅 store 状态，不直接维护冗余状态。更新时我会先更新 store，再触发 UI 响应式更新。对于异步请求，我会加 loading 状态和错误状态，避免 UI 显示与真实数据不一致。

在需要“乐观更新”的场景，比如删除文件，我会先在 UI 层做预更新，失败再回滚，这样用户体验更好。

### 组件与页面

1. 聊天页面的核心组件如何拆分？
A（5分钟版）：我把聊天页面拆成四部分：会话列表、消息列表、输入区、状态提示。会话列表管理历史会话切换；消息列表渲染用户和模型消息；输入区负责发送与停止；状态提示显示加载和错误。

在消息渲染上，我会将单条消息拆成独立组件，处理 Markdown、公式与代码高亮。这样消息渲染逻辑集中，便于维护。

此外，我还做了“流式输出”处理：后端返回的片段实时拼接到最后一条消息上，保证对话体验自然。

2. 文档上传页面的交互流程是什么？
A（5分钟版）：上传流程是“选择文件→计算 MD5→分片切割→并发上传→合并→状态轮询”。前端在本地先计算 fileMd5，然后按固定大小切片并并发上传。

上传过程中我会展示实时进度和速度，如果失败就提示重试。所有分片上传完成后调用合并接口，合并成功再更新列表。

3. 如何处理大文件分片上传的进度显示？
A（5分钟版）：进度显示由两部分组成：前端已完成分片数 / 总分片数，以及后端 status 接口返回的已上传分片列表。

我会把分片上传状态存本地变量，同时定期与后端状态对齐，避免因网络中断导致进度不准确。对于用户感知，我会展示“已上传百分比、当前上传速度、剩余时间估计”。

如果发生失败，我会对失败分片进行重试，并在 UI 上标记失败原因，让用户知道如何处理。

4. 如何渲染富文本（Markdown、公式、高亮）？
A（5分钟版）：我用 Markdown-it 解析消息文本，再结合 KaTeX 渲染公式，Highlight.js 渲染代码高亮。为了安全，我会做 XSS 过滤，确保不会渲染恶意脚本。

在性能上，我会对长文本做折叠，并避免一次性渲染过多内容。对于频繁更新的流式文本，我会做节流，减少 DOM 更新次数。

这样既保证显示效果，也保证性能稳定。

5. 如何优化长列表与历史记录展示性能？
A（5分钟版）：我主要用分页和虚拟滚动两种方式。对于历史记录列表，我会分页加载；对于聊天消息列表，我会使用虚拟滚动只渲染可视区域。

同时我会在切换会话时缓存已加载的数据，避免重复请求。对于搜索类列表，我会做防抖，减少无效请求。

这些优化能显著提升长列表的响应速度与体验。

重连时会重新鉴权，避免 token 过期导致一直失败。如果多次重连失败，则提示用户“连接异常”，允许手动重连。

对于复杂表格或长列表，我会在移动端改成卡片式布局，减少横向滚动。还会测试常见机型，确保交互稳定。

这样即使在移动端，也能保证核心功能可用。

## 前后端协作与接口设计（5分钟完整版，替代前文简答）

1. 如何定义统一的响应结构与错误码？
A（5分钟版）：我和后端约定统一返回结构为 `code/message/data/traceId`，其中 `code` 表示业务状态，`message` 用于提示，`data` 是业务数据，`traceId` 便于排查。

错误码遵循“HTTP 状态 + 业务码”结合的方式，前端根据错误码做统一处理，比如 401 触发刷新、403 提示无权限、500 提示系统异常。

这样前后端约定清晰，前端处理逻辑也能统一。

2. 文档上传的分片协议如何设计？
A（5分钟版）：前端先计算 `fileMd5`，然后按固定 `chunkSize` 切片。每个分片请求携带 `fileMd5`、`chunkIndex`、`totalChunks`、`fileName` 等字段。

上传前先调用 `status` 接口获取已上传分片，避免重复。分片上传完成后调用 `merge` 接口合并。整个流程是幂等的，重复上传不会破坏结果。

这样设计保证了断点续传和高并发情况下的稳定性。

3. 如何保证权限模型在前后端一致？
A（5分钟版）：我的原则是“后端强校验，前端只做 UI 控制”。后端在接口层和数据层都强制校验权限，前端只是根据角色和权限隐藏按钮或菜单。

为了保证一致性，我会从后端获取权限列表或角色信息，前端基于这些数据生成菜单与路由，避免前端硬编码。

这样即使前端逻辑出错，也不会影响数据安全。

4. 哪些接口适合幂等设计？
A（5分钟版）：分片上传、合并、删除、刷新 token、获取文件状态都适合幂等。因为这些接口可能会被重试，如果不幂等会导致数据重复或错误。

我在设计时会用 `fileMd5` 或 `requestId` 作为幂等键，后端检测后决定是否重复处理。

幂等设计能显著提升系统稳定性，尤其在网络抖动时非常重要。

5. 如何进行接口文档维护与版本控制？
A（5分钟版）：我采用 `/api/v1` 作为版本前缀，任何破坏性升级会新开 `/api/v2`，旧版继续维护一段时间。

文档维护上，我会使用 OpenAPI/Swagger 自动生成文档，并在变更时更新说明和示例。前端对接时会锁定版本，避免后端变更导致兼容问题。

这种版本化策略能够让系统持续迭代，同时保证旧功能稳定。

## 追加：5分钟扩展脚本（可直接背诵）

### 后端-项目整体口述脚本
A（扩展）：这是我自己的项目，所以整体架构和关键链路都是我亲自设计和实现的。我把系统拆成“上传与解析、检索与对话、权限与安全、运维与可观测”四条主线。上传与解析走分片上传 + MinIO 合并 + Kafka 异步解析；检索与对话走 ES 混合检索 + RAG 召回 + WebSocket 流式回复；权限与安全用 JWT + 组织标签 + 资源级过滤；运维与可观测用日志埋点、失败补偿和任务重试。这样拆分后，每个模块边界清晰，故障也容易定位。

具体链路上，我把上传与解析设计成强幂等流程：前端计算 fileMd5，后端以 fileMd5+userId 做唯一键约束；分片状态写 Redis bitmap，避免重复上传；合并成功后写入 file_upload 元数据并投递 Kafka。消费者解析文档后做“父块+子块”的切分，再做向量化，写入 ES 和 document_vectors，确保既能检索又能追踪来源。对话模块在检索后拼上下文，添加来源标注与限制规则，降低幻觉。我可以清晰讲出每个阶段的输入、输出、失败处理和日志点，这也是我把它当成一个工程化系统来做的证明。

在性能与可靠性上，我的关键取舍是“把重任务异步化，把权限下沉到数据层”。异步化保证接口快速返回，用户体验更好；权限下沉保证即使 controller 漏校验，查询结果也不会越权。整个项目的目标不是只实现功能，而是做到“可复用、可排障、可扩展”。

### 后端-安全与权限口述脚本
A（扩展）：安全这块我主要做了三层：身份认证、角色授权、资源隔离。身份认证用 JWT，Access Token 用于请求鉴权，Refresh Token 用于续期；为了做到主动失效，我引入 Redis 做 token 状态与黑名单。角色授权由 Spring Security 配置，普通用户与管理员权限清晰分层。资源隔离则是核心，我用 orgTag 作为组织隔离标签，并且把过滤逻辑下沉到查询层，在 MySQL、ES 查询时都加上 orgTag/userId/isPublic 的过滤条件。

我之所以强调“查询层强校验”，是因为任何只依赖前端或 controller 的权限判断都可能被绕过。我的实现中，所有涉及资源读取的接口都必须经过数据层过滤，即使有人构造请求，也拿不到不属于自己的数据。另外在 WebSocket 上我也做了鉴权和 stop token 的防护，防止恶意用户伪造指令中断他人会话。

在审计方面，我在登录、上传、下载、删除这些敏感操作上都有日志记录，这样出现权限问题可以快速追溯。整体上，这套权限模型是我自己从零设计并调试过的，知道每一层的边界和风险点。

### 后端-数据与缓存口述脚本
A（扩展）：数据层我用 MySQL 记录元数据与业务关系，用 Redis 做状态与缓存。MySQL 里我把 file_upload 作为主表，document_vectors 和 chunk_info 作为明细表，一对多关系清晰。最常见的查询是“按 fileMd5 / userId / orgTag 查文件列表”以及“按 fileMd5 查向量块”，所以我专门为这些场景建了组合索引。对于 org_tags 这种多值字段，我知道用逗号存储不够规范，所以我在文档里给出了更标准的 user_org_tags 关联表方案。

Redis 主要做四类数据：上传进度、token 状态、会话缓存、组织标签缓存。上传进度用 bitmap 是为了节省空间且快速判断分片状态；token 状态避免 JWT 无法主动失效的问题；会话缓存加 TTL 限制内存；组织标签缓存加快权限判断。缓存一致性上我坚持“数据库为准”，更新数据库后清缓存，读缓存未命中再查库回填，保证权限数据不会长期错误。

### 后端-检索与 RAG 口述脚本
A（扩展）：检索这块我做的是混合检索。用户问题先向量化，再在 ES 做 KNN 召回，保证语义相关；同时做关键词匹配和 rescore，提高精度。我会对召回结果做权限过滤，保证只有“本人、组织内、公开”三类资源可见。随后对文本块做长度截断和来源标注，拼成上下文，作为系统提示输入模型。

我还特别关注“可追溯性”，所以每个文本块都能映射到来源文件与段落编号。这样模型回答时即使有偏差，我也能快速定位到召回的原文，调参和排障效率很高。模型输出时我还会加入“只基于检索结果回答”的规则，减少幻觉。

### 后端-性能与运维口述脚本
A（扩展）：性能优化我主要围绕三点做：异步化、批量化、限流。解析和向量化通过 Kafka 异步，避免接口阻塞；ES 写入用 bulk，向量化用批量请求；在高并发场景下对上传和对话接口做限流与队列保护。可观测性方面我加了关键日志点，结合 Kafka 监控和死信队列定位失败任务；Redis 和 ES 也有指标监控，确保瓶颈可视化。

在一致性上，我采用最终一致性策略。删除文档时先删 ES，再删 MinIO，最后删 MySQL，任何一步失败都会记录日志并进入补偿队列。这样即使跨系统无法事务，也能保证最终收敛。对我来说，这些工程化设计是项目能稳定跑起来的关键。

### 前端-核心功能口述脚本
A（扩展）：前端我用 Vue3 + Vite + Pinia，主要是开发效率高、配置简单。我按模块拆分 API 和页面，登录、上传、检索、对话分别成模块，配合统一的请求封装与错误处理。分片上传这块，我在前端负责计算 fileMd5、切片、并发上传和进度统计；失败后只补传缺失分片，不会从头再传。对话页面用 WebSocket 流式接收，配合停止按钮与加载动画，保证体验流畅。

工程化上我使用环境变量区分环境，API 封装统一加 token 并做刷新逻辑，拦截器对错误码统一提示。路由守卫根据用户信息注入动态路由，保证权限可控。整体上前端是配合后端安全策略实现的，而不是单独的“页面展示”。

## 追加：更多场景题（详细版）

1. 场景：向量化服务经常超时，导致 Kafka 消费积压。
A（详细）：我会先看向量化接口的响应时间与失败比例，判断是网络问题还是模型服务本身性能不足。短期方案是调小批量、增加重试间隔，避免雪崩；中期方案是扩容向量化服务或切换更轻量模型；长期方案是把向量化任务做分级处理，优先处理热门文件或小文件，降低整体等待。必要时我会把向量化拆成独立服务并做限流。

2. 场景：ES 索引迁移时线上必须不停机。
A（详细）：我会采用“新建索引 + 双写 + 切别名”的方案。先建新索引并写入新数据，同时老索引继续服务；同步历史数据后，用别名切换读流量，再停写老索引。这样迁移过程无停机，且可以快速回滚。

3. 场景：Redis 宕机导致上传进度丢失。
A（详细）：我会先保证核心功能可用，允许用户重新上传缺失分片。进度恢复可以通过 MinIO 列表接口获取已上传分片，再重建 bitmap。长期上我会给 Redis 做持久化和主从，避免单点。

4. 场景：Kafka 消费重复导致 ES 数据重复。
A（详细）：我会用 fileMd5 + chunkId 做幂等写入，ES 写入前先检查是否存在，或者用固定文档 ID 覆盖写。这样重复消费也不会产生多份数据。

5. 场景：上传后文件名相同但内容不同，导致用户混淆。
A（详细）：我会以 fileMd5 作为主唯一标识，前端展示时显示文件名 + 上传时间或版本号。必要时允许用户改名或做版本管理。

6. 场景：用户要求批量删除文档，但部分删除失败。
A（详细）：批量删除时我会返回每个文件的删除结果，同时失败的记录进入补偿队列。前端展示失败清单，允许用户重试或联系管理员。

7. 场景：权限变更后用户仍能访问旧资源。
A（详细）：这是缓存一致性问题。我会在权限变更时清理 Redis 里的 orgTag 缓存，并在 ES 与 MySQL 查询层强制过滤，确保即使缓存没更新也不会越权。

8. 场景：模型回答出现明显幻觉。
A（详细）：我会首先看召回是否准确，如果召回文本不相关就调检索参数；其次检查 prompt 规则是否足够严格；必要时增加“引用原文”的约束，并降低回答温度。对高风险问题我会提示用户“无相关资料”。

9. 场景：解析后文本乱码或丢失。
A（详细）：我会检查文件编码识别是否正确，必要时针对常见格式增加专用解析器。对扫描 PDF 可能要做 OCR，或者提示用户转换格式。

10. 场景：用户投诉检索结果太慢。
A（详细）：我会检查 ES 查询耗时和向量化耗时，确认瓶颈在召回还是模型。短期可以降低 topK 或做缓存；长期可以升级 ES 集群或做冷热数据分层。
